# e-SOSTAC Founder Seeding Sprint

Use this playbook to seed Wasel deliberately, starting with one corridor and one
core use case before wider expansion.

## Scope

- Sprint length: `14 days`
- Primary corridor: `Amman <-> Zarqa`
- Backup corridor: `Amman <-> Irbid`
- Primary offer: `shared rides`
- Secondary offer: `same-route packages`
- Operating model: `manual trust + assisted matching + low-cost community seeding`

## e-SOSTAC

### Situation

- The app is now credible enough to support controlled seeding.
- The biggest launch risk is corridor liquidity, not UI quality.
- Early wins will come from building visible route activity in a narrow area.
- Manual support is an advantage at this stage because it increases trust.

### Objectives

Hit these targets in the first `14 days`:

- `75` qualified users onboarded
- `20` supply-side users activated
- `40` demand-side users complete at least one route search
- `10` successful ride matches
- `3` package requests
- `20%` week-2 repeat usage among activated users
- `15%` referral participation among active users
- `<= 10 min` first-response time for onboarding/support

Hit these targets in the first `30 days`:

- `150` qualified users onboarded
- `40` supply-side users activated
- `80` demand-side users complete at least one route search
- `25` successful ride matches
- `8` package requests
- `20%+` repeat usage
- `15%+` referral participation

### Strategy

- Seed one corridor until it feels alive.
- Bias toward trust, responsiveness, and proof-of-use over reach.
- Build both supply and demand manually at first.
- Turn WhatsApp and small communities into the first distribution engine.
- Expand only after the first corridor shows repeatable matching.

### Tactics

- Channels:
- WhatsApp groups
- university and college communities
- neighborhood and commuter Facebook groups
- driver micro-networks
- small businesses with repeated package movement
- one-to-one outreach from trusted operators

- Hooks:
- cheaper than solo transport
- shared route, clearer timing
- trusted WhatsApp support
- packages move on the same corridor as people
- founder access and fast response

- Offers:
- first completed ride reward
- driver founding badge
- referral credit after first completed trip
- priority support for first 100 users

### Action

#### Week 1

Day 1:
- confirm corridor owner and support owner
- create a daily WhatsApp operating rhythm
- prepare founder list of `20` driver or route-poster candidates
- prepare founder list of `50` rider candidates

Day 2:
- recruit first `5` trusted drivers or ride posters
- onboard them manually
- verify route hours, preferred pickup points, and willingness to accept packages

Day 3:
- recruit first `15` riders from the primary corridor
- guide each rider through search, save, and booking intent
- log every objection in the tracker

Day 4:
- post founder message into corridor groups
- follow up 1:1 with every high-intent reply
- push riders to search the corridor on the same day

Day 5:
- publish first route availability window
- manually coordinate the best driver-passenger matches
- capture first screenshots/testimonials if any ride completes

Day 6:
- recruit next `5` supply-side users
- contact `10` small merchants or resellers for package trial interest
- push "same corridor, same day" package use case

Day 7:
- review week-one numbers
- cut low-quality channels
- identify the top `3` objections blocking conversion

#### Week 2

Day 8:
- launch referral push to active users
- ask every successful rider and driver for one introduction

Day 9:
- re-engage everyone who searched but did not book
- send route-specific reminder, not a generic promo

Day 10:
- run a package mini-pilot with `2-3` senders on the corridor
- verify tracking and support responsiveness

Day 11:
- publish proof-of-use:
- completed trips
- support speed
- corridor activity windows

Day 12:
- strengthen repeat behavior
- prompt riders to save recurring routes
- prompt drivers to post next-day availability

Day 13:
- score the corridor:
- supply density
- search volume
- matches
- repeat behavior
- support burden

Day 14:
- make the expansion decision
- if the corridor is healthy, prepare backup corridor seeding
- if not, stay focused and improve liquidity on the first corridor

### Control

Review these metrics every day:

- new qualified users
- activated supply users
- activated demand users
- searches
- posts or offered routes
- completed matches
- package requests
- repeat users
- referrals used
- support response time
- top objections

Use these rules:

- If searches are growing but matches are weak, increase supply quality.
- If supply exists but bookings are weak, improve trust and follow-up.
- If first-time users convert but do not return, fix retention messaging.
- If support load becomes chaotic, narrow the corridor even more.
- Do not open the backup corridor unless the primary corridor shows momentum.

## Daily Standup

Use this format every day:

1. What changed in corridor liquidity yesterday?
2. How many riders searched?
3. How many drivers posted or confirmed availability?
4. How many real matches happened?
5. Which objections appeared most often?
6. Which channel produced the strongest users?
7. What one change will we make today?

## Seeding Roles

- Corridor owner: owns daily volume and conversion
- Supply owner: recruits drivers and route posters
- Demand owner: recruits riders and re-engages searches
- Support owner: handles onboarding and issue response
- Evidence owner: collects testimonials and proof-of-use

## Expansion Gate

Do not seed the backup corridor until all of the following are true for the
primary corridor:

- `>= 10` successful ride matches
- visible daily search activity
- at least `5` repeat users
- at least `5` reliable supply-side users
- support remains manageable

## Message Discipline

- Talk about one corridor, not the whole country.
- Talk about today or tomorrow's route availability, not abstract future value.
- Talk about trust, clarity, and support before talking about scale.
- Use screenshots and specific examples whenever possible.
